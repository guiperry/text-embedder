1. Executive Summary

The text-embedder is structurally well-suited for serverless deployment. Its core embedding algorithm is already stateless — deterministic, side-effect-free, and backed by zero external dependencies. The only obstacles are an architectural layer: the project is currently packaged as a long-running HTTP server (with goroutine worker pools, graceful shutdown, and a CLI flag parser) rather than as discrete serverless function handlers.

The refactor has one critical performance fix, one structural restructure, and platform-specific wiring. No algorithm changes are required. Estimated effort: 1–2 days for a careful implementation including tests.

Good News
The pkg/embed package requires zero changes to its public API. All consumers (the SocratiCode provider, importers, etc.) keep the same import path and function signatures.

2. Current Architecture Analysis

What the server does today

cmd/text-embedder/main.go is a long-lived process that registers four routes on a http.ServeMux and runs until it receives SIGINT/SIGTERM. It uses a global batchWorkers integer (set from a CLI flag at startup) to bound the goroutine pool inside handleBatch.

3. Stateless Blockers & Constraints

Critical — Landmark Vector Recomputation

pkg/embed/embed.go:66 calls getLandmarkVectors() on every single call to Embed(). That function iterates all 768 landmark strings through the full embedToLattice() pipeline (tokenize → hash → accumulate over a 4096-dim lattice). For a batch of 256 texts that is 256 × 768 = 196,608 landmark vectorizations per request — on top of the 256 vectorizations for the actual inputs. This will hit Vercel's default 10-second function timeout on large batches.

Fix: Wrap getLandmarkVectors() with a sync.Once so the landmark matrix is computed exactly once per process lifetime (i.e., once per warm container). The computation drops to zero on all subsequent invocations within the same warm function instance.

Warning — Batch Timeout Risk

Vercel's Hobby plan caps function execution at 10 seconds; Pro at 60 seconds; Enterprise at 900 seconds. Even with the sync.Once fix, a batch of 256 texts on a single-vCPU cold container may approach the Hobby limit. Recommend capping the serverless batch size at 128 (configurable via a separate constant) and documenting that the local binary supports up to 256. The vercel.json maxDuration should be set to 60 to use the Pro limit safely.

Warning — Worker Pool Assumption

The current batch handler creates a semaphore sized to GOMAXPROCS, set at startup via the --workers flag. Serverless functions typically run with GOMAXPROCS=1 or 2 and have no startup phase. The resolution chain must therefore be: (1) --workers flag (binary only, preserves backward compatibility with existing providers and agentic callers) → (2) EMBED_BATCH_WORKERS env var (Vercel dashboard / CI) → (3) runtime.GOMAXPROCS(0) as the universal fallback. This means existing providers that already pass --workers see no change whatsoever, while the Vercel deployment gains a tuneable escape hatch via env var.

Note — No Database, No Sessions, No Shared State

The embedding algorithm has no database writes, no session cookies, no in-memory caches that need to be shared across instances, and no file system writes. Every request is truly self-contained. This makes the refactor substantially simpler than a typical stateless migration.

4. Why Vercel

Vercel is the deployment target for this refactor. Its Go runtime requires no additional SDK or dependencies — any .go file placed in an api/ subdirectory is automatically compiled and deployed as a serverless function. Route rewrites in vercel.json let the existing API surface (/embed, /embed/batch, /similarity, /health) remain intact so all current clients need zero reconfiguration.

5. Step-by-Step Refactor Plan

  1. Fix: Add sync.Once to getLandmarkVectors()

    In pkg/embed/embed.go, add a package-level sync.Once and a cached [][]int64. The first call to Embed() after a cold start pays the computation cost once (~5–15ms for 768 landmarks); all subsequent calls within the same warm function container return the cached slice instantly. This change is backward compatible and safe for concurrent callers.

  2. Create api/ handler files for Vercel

    Extract each of the four route handlers from cmd/text-embedder/main.go into separate files under api/. The handler logic is identical — only the packaging changes. Each file must export a function named Handler with the signature func Handler(w http.ResponseWriter, r *http.Request) in package handler (Vercel's convention for Go functions).

  3. Extract shared types & helpers into an internal package

    The request/response structs (embedRequest, batchResponse, etc.) and helpers (writeJSON, writeError, decode) are duplicated across all four handlers if left inline. Move them to internal/api/helpers.go so all handler files share them without copying.

  4. Implement flag → env var → GOMAXPROCS worker resolution

    The --workers CLI flag is preserved in cmd/text-embedder/main.go for full backward compatibility with existing providers and agentic callers. A shared resolveWorkers(flagVal int) int helper in internal/api encodes the three-tier chain: non-zero flagVal wins; otherwise parse EMBED_BATCH_WORKERS env var; otherwise fall back to runtime.GOMAXPROCS(0). The binary passes its flag value in; the Vercel handler passes 0 (no flag), so it naturally falls through to the env var or GOMAXPROCS. No existing caller is broken.

  5. Add CORS headers

    Serverless APIs are typically called from browser contexts. Add permissive CORS headers (Access-Control-Allow-Origin: *) to all handlers and handle OPTIONS preflight requests. This can be done centrally in the shared helpers or via vercel.json headers config.

  6. Add vercel.json

    Configure route rewrites so that /embed maps to /api/embed, preserving the existing API surface exactly. Clients (including the SocratiCode provider) need zero configuration changes. Set maxDuration: 60 for the batch function to avoid Hobby-tier timeouts on large batches.

  7. Create public/index.html — API documentation landing page

    Vercel automatically serves the public/ directory as static assets, making public/index.html the page cloud users land on when they visit the root URL. The page should document every endpoint (method, path, request schema, response schema, and a working curl example), show the Go import usage, list the model constants (ModelID, Dims), and link to the GitHub repo. This replaces ad-hoc README reading for cloud consumers who never clone the repo.

  8. Keep cmd/text-embedder/ for local development

    The existing long-running server binary is still useful for local development, benchmarking, and as the production binary for self-hosted deployments. Update it to import handlers from the new shared location instead of duplicating logic. Add a note to the README documenting both deployment modes.

  9. Update go.mod module path and verify build

    Run go build ./... and go test ./... to confirm nothing is broken. The Vercel CLI can also do a local preview build with vercel dev before any deployment.

  10. (Optional) Reduce serverless batch cap to 128

    The current limit of 256 was chosen for the self-hosted binary where CPU time is unconstrained. For the serverless handler, add a separate constant (e.g. serverlessBatchLimit = 128) to stay comfortably within the 10s Hobby timeout. Leave the 256 limit in the local binary unchanged.

7. Key Code Transformations

7a. sync.Once landmark cache in pkg/embed/embed.go

The most impactful single change. Replace the inline getLandmarkVectors() call inside Embed():

// BEFORE (embed.go:66) — recomputes 768 lattice vectors on every Embed() call
func Embed(text string) []int32 {
    lat := embedToLattice(text)
    landmarks := getLandmarkVectors()   // ← hot path, 768 full vectorizations
    ...
}
// AFTER — compute once, reuse forever within the warm container
var (
    landmarkOnce    sync.Once
    landmarkVectors [][]int64
)

func cachedLandmarkVectors() [][]int64 {
    landmarkOnce.Do(func() {
        landmarkVectors = getLandmarkVectors()
    })
    return landmarkVectors
}

func Embed(text string) []int32 {
    lat := embedToLattice(text)
    landmarks := cachedLandmarkVectors()   // ← O(1) after first call
    ...
}

7b. Vercel handler file (api/embed.go)

Vercel's Go runtime discovers handler files by looking for an exported Handler function in package handler. The handler logic is identical to today's handleEmbed — only the packaging differs. 

package handler

import (
    "net/http"

    api "github.com/guiperry/text-embedder/internal/api"
    "github.com/guiperry/text-embedder/pkg/embed"
)

func Handler(w http.ResponseWriter, r *http.Request) {
    api.SetCORSHeaders(w)
    if r.Method == http.MethodOptions {
        w.WriteHeader(http.StatusNoContent)
        return
    }
    if r.Method != http.MethodPost {
        api.WriteError(w, http.StatusMethodNotAllowed, "method not allowed; use POST")
        return
    }
    var req api.EmbedRequest
    if !api.Decode(w, r, &req) {
        return
    }
    vec := embed.Embed(req.Text)
    api.WriteJSON(w, http.StatusOK, api.EmbedResponse{
        Embedding:  vec,
        Dimensions: embed.Dims,
        Model:      embed.ModelID,
        TokenCount: len(embed.Tokenize(req.Text)),
    })
}

7d. resolveWorkers — flag → env var → GOMAXPROCS chain

This helper lives in internal/api/helpers.go and is shared by both the binary and the serverless handlers. The binary passes its parsed --workers flag value; the Vercel handlers pass 0, which falls through to the env var or GOMAXPROCS. 

// resolveWorkers returns the concurrency bound for the batch semaphore.
// Priority: --workers flag (binary only) → EMBED_BATCH_WORKERS env var → GOMAXPROCS
func ResolveWorkers(flagVal int) int {
    if flagVal > 0 {
        return flagVal   // --workers flag wins; preserves existing provider/agentic behavior
    }
    if s := os.Getenv("EMBED_BATCH_WORKERS"); s != "" {
        if n, err := strconv.Atoi(s); err == nil && n > 0 {
            return n   // env var wins for serverless deployments
        }
    }
    return runtime.GOMAXPROCS(0)   // universal fallback
}
// cmd/text-embedder/main.go — unchanged flag registration; passes flagVal to handler
workers := flag.Int("workers", runtime.GOMAXPROCS(0), "max concurrent batch workers")
flag.Parse()

mux.HandleFunc("/embed",       api.HandleEmbed)
mux.HandleFunc("/embed/batch", func(w http.ResponseWriter, r *http.Request) {
    api.HandleBatch(w, r, api.ResolveWorkers(*workers))
})
mux.HandleFunc("/similarity",  api.HandleSimilarity)
mux.HandleFunc("/health",      api.HandleHealth)
// api/embed/batch.go — Vercel handler; passes 0 so flag falls through to env/GOMAXPROCS
func Handler(w http.ResponseWriter, r *http.Request) {
    api.SetCORSHeaders(w)
    if r.Method == http.MethodOptions { w.WriteHeader(http.StatusNoContent); return }
    api.HandleBatch(w, r, api.ResolveWorkers(0))
}

Observability Structured slog output to stdout Platform captures stdout/stderr; custom logging middleware not needed
Cost model Fixed cost of server/VPS Pay-per-invocation; very cheap for low/moderate traffic, can be expensive under sustained high load

Provider Queue & Binary Lifecycle
The provider-example.js file assumes a locally-running binary with port probing and a kill/restart recovery loop. When the provider points at a Vercel URL instead, the binary lifecycle management (resolveBaseUrl, killBinary, probePort) becomes irrelevant — the URL is always available. The provider should support a TEXTEMBEDDER_URL env var pointing to the deployed Vercel URL so the binary-spawn path is bypassed entirely.
